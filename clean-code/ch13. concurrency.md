## 동시성과 관련된 미신
1. 동시성은 항상 성능을 높여준다 
	=> 대기 시간이 아주 길어 여러 스레드가 프로세스를 공유할 수 있거나 여러 프로세서가 처리할 독립적인 계산이 충분히 많은 경우에만...
2. 동시성을 구현해도 설계는 변하지 않는다
	**무엇**과 **언제**를 분리하면 설계가 많이 변한다. 

## facts...
- 다소 부하를 요구함
- 구현이 복잡하고 동시성 버그는 재현하기가 어렵다
- 동시성을 구현하려면 근본적인 설계 전략을 재고해야 함

## 동시성 방어 원칙
1. SRP: 동시성 관련 코드는 다른 코드와 분리할 것
2. 임계영역(critical section)을 보호하라 -> 스레드 간 공유로 인해 예상치 못한 결과가 생김
3. 임계영역을 할 바엔 차라리 읽기 전용으로 객체 복사를 하는 것도 좋음
4. 스레드는 가능한 독립적으로 구현할 것: 각자의 자원만을 사용할 것, 자료를 독자적으로 사용가능하게 분리할 것

# 패턴
## Producer-Consumer
- **생산자(Producer)**: 작업을 만들어서 버퍼(큐)에 넣는다.
- **소비자(Consumer)**: 큐에서 작업을 꺼내 처리한다.
-  생산자는 큐에 작업물을 넣고 소비자에게 "큐가 비어있지 않다"는 신호를 보내고 소비자는 큐에서 작업물을 꺼낸 후 "큐가 가득차 있지 않다"는 신호를 보낸다. 
- 그 전까지 둘은 신호를 기다린다.

solution: 
- semaphore: 빈 슬롯과 아이템 개수를 세는 세마포어 + 뮤텍스
- blocking queue

## Readers-Writers
- **Reader**: 읽기만 하는 스레드 (다수 동시에 가능)
- **Writer**: 쓰기를 시도하는 스레드 (단독만 허용)
- 처리량을 강조해 독자가 상대적인 우선권을 가지게 되면 저자는 기아 상태에 빠지고 공유 자원은 정체된 정보로 가득차게 된다. 
- 반대로 저자가 우선권을 가지면 처리량이 줄어들게 된다.

solution:
- RW Lock: when read -> pass read & block write access
- policy: FIFO queue

## **Dining Philosophers**
- 여러 프로세스(철학자)가 둘씩 공유 자원(포크)을 써야 할 때
- 각 프로세스는 양쪽 자원을 모두 확보해야만 작업(식사)을 수행
- 자원을 잠글 때 잘못된 순서를 택하면 시스템 전체가 멈춤
	1. **Deadlock**: 전원이 왼쪽 포크만 잡고 서로 기다리면 영원히 멈춤 
	2. **Livelock**: 서로 양보만 반복 → 실행 안 됨
	3. **Starvation**: 특정 철학자는 계속 굶주림
	4. **비효율성**: 과도하게 보수적 설계 → 병렬성 낮아짐

근본적인 원인: 포크를 잡을 때 atomic 하게 안 잡아서 그럼

solution:
- **Resource hierarchy (Ordered locks)**: 포크에 순서를 매겨 일관되게 획득(hashing)
- **타임아웃 + 재시도**: 일정 시간 후 락 재시도하여 교착 방지

요약
1. 쓸데없는 구간을 과하게 잠구지 말고 잠가야 할 필요가 있는 부분은 잠근다. 
2. 공유 객체와 공유 구간을 최소한으로 줄여라
3. 보조 코드 등을 추가하여 무작위 순서로 실행되는 것을 여러번 테스트
