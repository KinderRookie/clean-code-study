# 1. 플래그 인수는 추하다
```csharp
public void SaveFile(string path, bool overwrite)
{
    if (overwrite)
        OverwriteFile(path);
    else
        CreateNewFile(path);
}
```

## **❌ 플래그 인수가 추한 이유**
### **1.** **함수가 두 개의 일을 한다** **→** **SRP 위반**
플래그가 있다 = 내부에 if (flag) 로직이 있다 = 두 가지 행동이 있다.

➡ 하나의 함수가 **한 책임(SRP)** 을 어겨서,
➡ 함수 읽는 사람은 “지금 이 flag가 true인지, false인지” **계속 추론해야 함**.
### **2. **추상화 수준이 섞인다**
- SaveFile은 고수준의 이름인데,
- 내부는 overwrite냐 아니냐에 따라 **로우레벨 분기**가 생김.
- → 함수 내에서 **여러 수준의 세부사항이 공존** → **내려가기 규칙 위반**


# 2. side-effects
```csharp
public class UserValidator

{
    private readonly Cryptographer _cryptographer;

    public UserValidator(Cryptographer cryptographer)
    {
        _cryptographer = cryptographer;
    }

    public bool CheckPassword(string userName, string password)

    {
        var user = UserGateway.FindByName(userName);
        if (user != User.Null) 
        {
            string codedPhrase = user.GetPhraseEncodedByPassword();
            string phrase = _cryptographer.Decrypt(codedPhrase, password);

            if (phrase == "Valid Password")
            {
                Session.Initialize();
                return true;
            }
        }
        return false;
    }
}
```
`Session.Initialize();` 부분에서 side-effect 발생.
- 의도: 유저 인증
- side-effect: 세션이 초기화됨
- temporal coupling 문제: 순서가 중요한 경우!
	- 즉, 초기화를 해도 되는 상황이 보장된 경우에만 호출 가능


# try-catch 블록 뽑아내기
```csharp
public void delete(Page page) {
	try {
		deletePageAndAllReferences(page);
  	} catch (Exception e) {
  		logError(e);
  	}
}

private void deletePageAndAllReferences(Page page) throws Exception { 
	deletePage(page);
	registry.deleteReference(page.name); 
	configKeys.deleteKey(page.name.makeKey());
}

private void logError(Exception e) { 
	logger.log(e.getMessage());
}
```
