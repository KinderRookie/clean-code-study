# 시스템 제작과 사용의 분리
```Java
  public Service getService() {
      if (service == null)
          service = new MyServiceImpl(...);
      return service;
  }
```
문제점: 구현과 런타임 로직이 섞여 있다. `getService()` 메서드가 `MyServiceImpl` 객체의 생성자 인수에 의존한다. service가 null인 경우와 아닌 경우 모두 테스트 해봐야 한다. 

**설정 논리는 일반 실행 논리와 분리해야 한다.**

방법들
1. Main 분리 방법
   생성과 생성과 관련된 코드는 main이나 main이 호출하는 모듈로 옮기고 나머지 시스템은 모든 객체가 생성되었고 모든 의존성이 해결되었다고 가정한다. 
   [![](https://github.com/Yooii-Studios/Clean-Code/raw/master/images/figure%2011-1.png)](https://github.com/Yooii-Studios/Clean-Code/blob/master/images/figure%2011-1.png)
2. 팩토리 분리 방법
   객체가 생성되는 시점을 어플리케이션이 결정하는 기법. 그러나 여전히 객체가 생성되는 구체적인 방법을 모른다. 
   [![](https://github.com/Yooii-Studios/Clean-Code/raw/master/images/figure%2011-2.png)](https://github.com/Yooii-Studios/Clean-Code/blob/master/images/figure%2011-2.png)
3. 의존성 주입
   객체는 의존성 자체를 인스턴스로 만드는 책임을 지지 않으며, 이런 책임을 다른 '전담' 매커니즘에 맡겨야 한다. 대신 의존성을 주입하는 방법으로 setter 나 생성자 인수를 제공한다.
   진정한 의존성 주입은 클래스가 해결하는 것이 아닌 DI 컨테이너가 요청이 들어올 때 마다 필요한 객체의 인스턴스를 만든 후 의존성을 설정하고 객체를 반환 한다. 

# scaling

```csharp
public interface IBank
{
    string GetStreetAddr1();
    string GetStreetAddr2();
    string GetCity();
    string GetState();
    string GetZipCode();
    void SetStreetAddr1(string street1);
    void SetStreetAddr2(string street2);
    void SetCity(string city);
    void SetState(string state);
    void SetZipCode(string zip);
    ICollection GetAccounts();
    void SetAccounts(ICollection accounts);
    void AddAccount(AccountDTO accountDTO);
}
```

```csharp
public class Bank : IBank
{
    private string _streetAddr1;
    private string _streetAddr2;
    private string _city;
    private string _state;
    private string _zipCode;
    private ICollection _accounts = new ArrayList();

    public string GetStreetAddr1() => _streetAddr1;
    public string GetStreetAddr2() => _streetAddr2;
    public string GetCity() => _city;
    public string GetState() => _state;
    public string GetZipCode() => _zipCode;

    public void SetStreetAddr1(string street1) => _streetAddr1 = street1;
    public void SetStreetAddr2(string street2) => _streetAddr2 = street2;
    public void SetCity(string city) => _city = city;
    public void SetState(string state) => _state = state;
    public void SetZipCode(string zip) => _zipCode = zip;

    public ICollection GetAccounts() => _accounts;
    public void SetAccounts(ICollection accounts) => _accounts = accounts;

    public void AddAccount(AccountDTO accountDTO)
    {
        // ⚠️ 이건 JNDI lookup 같은 역할 (직접 서비스 탐색 및 생성)
        var accountService = new AccountServiceFactory().Create();
        var account = accountService.Create(accountDTO);
        _accounts.Add(account);
    }

    // ⚠️ 이상한 생명주기 메서드들
    public void SetEntityContext(object ctx) { }
    public void UnsetEntityContext() { }
    public void Activate() { }
    public void Passivate() { }
    public void Load() { }
    public void Store() { }
    public void Remove() { }

    // ID 처리
    private int _id;
    public void SetId(int id) => _id = id;
    public int GetId() => _id;
    public void Create(int id) => _id = id;
}
```

**이것들의 문제점**
1. 비즈니스 로직이 컨테이너에 타이트하게 연결되어 있다. 객체를 생성하기 위해 컨테이너에서 제공해야 하는 인터페이스를 반드시 구현해야 한다는 뜻이다. 개발자가 비즈니스 로직과 관계없는 코드까지 억지로 작성해야 한다는 뜻이다. 
2. 테스트가 어렵고, 상속도 어렵고, 쓸데없이 자료를 저장하는 DTO도 작성하게 강제한다.

# cross-cutting 관심사
persistence, logging 등의 일부 동작(관심사)은 객체 경계를 넘나드는 경향이 있다. 원론적으로는 모듈화되고 캡슐화된 방식으로 영속성을 구상할 수 있으나, 현실은 코드 곳곳에 산재해 있을 가능성이 높다. 

**AOP의 도입**
AOP에서 관점이라는 모듈 구성 개념은 "특정 관심사를 지원하려면 시스템에서 특정 지점들이 동작하는 방식을 일관성 있게 바꾸어야 한다"고 명시함.
